# Tasks

Work briefs handed to the exec agent live here. Each file is a self-contained assignment — the exec agent reads only the brief and the docs it references, without access to the coordinator-level conversation.

## Convention

- **Filename:** `<slug>.md`, where the slug describes the work (e.g., `bootstrap-project-skeleton.md`, `seam-c-regex-strategy.md`, `pydantic-from-pydantic-impl.md`). Matches the output slug if the task produces a research doc (`docs/research/<matching-slug>.md`).
- **Template:** [`0000-template.md`](0000-template.md).
- **Structure:** Read first, Goal, Scope (numbered areas), Guardrails, Deliverable, Success criteria, Downstream consequences.
- **Self-contained:** every task brief assumes a cold reader. Link to every doc the agent needs. Don't assume the exec agent has context from prior conversation.

## Lifecycle

1. Coordinator drafts a brief in response to a decision point, a seam that needs implementation, or a gap surfaced by research.
2. User reviews and adjusts (or asks coordinator to adjust).
3. User relays the brief to the exec agent. The agent reads the brief + linked docs + does the work.
4. Output:
   - for implementation tasks → code + tests in the repo, per the success criteria
   - for research tasks → `docs/research/<matching-slug>.md`
5. Worker completion includes the standard evidence bundle from [`docs/process/evidence-bundle.md`](../process/evidence-bundle.md), with any lane-specific proof added by the brief.
6. Coordinator + user review the result and decide next actions (new task, new ADR, revise architecture, close thread).

## Task types

Match the label in the brief to one of these types:

- **`truth/data`** — inspect source material, artifacts, or runtime state to establish ground truth before formalizing
- **`contract/seam`** — define or tighten a seam contract; may update `docs/architecture.md`
- **`implementation`** — make an already-decided contract true in code, with contract tests
- **`observability/debug`** — add or tighten diagnosis surfaces (spans, replay fixtures, trace semantics)
- **`research`** — investigate a library, pattern, or prior art; output is a research doc, not code

See `docs/thread-orchestration.md` for the full thread-type taxonomy and when to pull a thread back to the coordinator.

## Why this directory exists

Without a dedicated place, task briefs clutter chat history. They're also worth versioning — the brief we *sent* is part of the record, not just the output it produced. Keeping briefs here means anyone (us, or a future collaborator) can see exactly what was asked, not just what was answered.
