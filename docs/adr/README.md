# ADRs

Architectural decision records for extractx. One file per decision, chronological numbering.

## Convention

- **Filename:** `NNNN-<slug>.md`, where `NNNN` is a zero-padded sequence number and the slug names the decision (e.g., `0001-evidence-engine-boundary.md`).
- **Template:** [`0000-template.md`](0000-template.md).
- **Status lifecycle:** `Proposed` → `Accepted` → `Superseded by ADR-NNNN` or `Deprecated`. An ADR is never deleted; superseded ones stay in the record.

## What goes in an ADR

Decisions that commit the architecture in a way that would be costly to reverse later:

- adopting or rejecting a significant library or ecosystem (pydantic, pydantic-ai, opentelemetry, msgspec)
- carving or renaming a seam
- changing a canonical object's shape or lifecycle
- changing a precedence rule
- dropping, deferring, or promoting an optional seam
- committing to a public-api surface decision (tier 1 vs tier 2 vs internal)
- any decision the architecture doc (`docs/architecture.md`) would need to be updated to reflect

Smaller decisions (implementation-local, refactor-local, "which helper to write") do not need an ADR.

## Relationship to the architecture doc

`docs/architecture.md` is the **current state** of the design. ADRs are the **history of how it got there**. When an ADR is accepted that changes the architecture, the architecture doc is updated in the same change so the two stay consistent.

If the architecture doc and an ADR disagree, the architecture doc wins for **what is true now**; the ADR wins for **why it got that way**. If they disagree about what should be true, that's a bug — resolve by writing a new ADR.

## Relationship to tasks

ADRs capture decisions. Tasks capture work briefs for the exec agent. An ADR may produce one or more follow-up tasks (to implement the decision). A task may surface findings that lead to a new ADR.
